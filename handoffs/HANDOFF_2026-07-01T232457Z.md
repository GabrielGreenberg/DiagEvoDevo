## Where we are
v1 (M0–M7) shipped, then the user redirected the **scoring philosophy** and we rebuilt the heart:

- **Comprehensive matrix score is now the default** (`config.scoring = 'comprehensive'`). Instead of
  collapsing each data relation to one carrier, the score compares each relation, like-with-like, against
  **all** its commensurable measurements and sums them **per-relation-normalized**:
  `reward = Σ_relations (Σ_measurements rewardValue) / relMax`, each relation ∈ [0,1], quality = reward/2.
  - Lattice change: reads-down chain is now `ordinal ≤ interval ≤ ratio ≤ cyclic` — **cyclic is the TOP**
    (a bearing from the reference carries ratio + order). So sales(ratio) → 20 measurements (15 ratio + 5
    cyclic); order(ordinal) → all 26. (`src/core/scale.ts`.)
  - `fRatio` got a positivity floor `log(max(c, eps.ratioPos))` (new gradchecked `maxConst` primitive) so
    signed carriers (run/rise/projections/bearings) score without NaN.
  - `Breakdown` is now `relations[]` with per-measurement fidelities + `normalized`; `scorePanel` shows the
    matrix + top carriers. `gradient`/`session` dropped the single-carrier map from the scoring path.
  - Result: **value is encoded richly** (several ratios track it) **AND order encodes** (best carrier
    ~0.97). A **common orientation emerges from the score itself** (parallel segments) — no penalty needed.
- **`fixed` mode** (`config.scoring = 'fixed'`) is retained as a swappable option: the earlier
  single-carrier bar-chart model (sales→length, order→x-position), which converges fast to clean bars.
- Nothing is mid-flight; everything committed and green.

The user's four choices that shaped this (do not re-litigate): (1) **sum** within a relation — more matches
wins; (2) **order readable from angles** → cyclic on top; (3) **normalize per relation** to balance
sales-vs-order; (4) **frame stays at the page origin** for now.

  **relative** plateau (`config.converge.plateauRelEps`), not just absolute. If you touch scoring scale,
  re-check convergence.
- **Perf knobs already dialed down** for the heavy score: `populationSize 8→4`, `stepsPerFrame 20→4`.
  Raise them back if you optimize the hot path.
- **Adversarial review paid off earlier** (M4): a zero-length segment used to NaN the whole 48-gradient;
  fixed by `eps.length`/`eps.ratioPos` floors, the `atan2`-at-origin guard, and the `circularVar`
  sign fix. Regression tests in `src/core/degeneracy.test.ts` — keep them.
- **Ordinal surrogate is spread-normalized + temperature-annealed** (`config.anneal`): margins are divided
  by `spread(c)` (scale-invariant) and T anneals `tStart→T` to give a global sorting force. Don't
  "simplify" it back to absolute margins — that breaks ordering (order got stuck at ~0.72 before).
- **Nomenclature** (panel labels): id = `anchor·part·reading`. `f·` = frame anchor (else page); `disp` =
  displacement (E−S); `∥`/`⊥` = proj parallel/perp to the reference direction; `mag` = length; `∠` = angle.
  `disp ∥` = run, `disp ⊥` = rise. `f·disp` == `disp` while the frame is parallel to the page.
- **Tests that are slow**: `src/optim/session.test.ts` runs a real convergence (~50 s under comprehensive)
  and has explicit timeouts; don't be surprised.
- All randomness flows through `src/core/rng.ts` (mulberry32) — results are reproducible from seeds.

## Standing principles — reproduced verbatim, every handoff (do not paraphrase, do not drop)

### I. Depth over patches
Add every feature at the deepest correct layer. If a change belongs in the architecture,
change the architecture — refactor rather than bolt on. A surgical patch that avoids a
warranted structural change is a defect, not a shortcut. When unsure whether a change is
deep or shallow, choose deep, even at the cost of a larger refactor now.

### II. Workflows over re-derivation
Build and verify through repeatable workflows — scripts, task runners, test suites — not by
re-reading files or re-deriving results inside the context window. Prefer invoking a
workflow to reconstructing knowledge. This holds for construction as much as verification.
If you find yourself doing the same multi-step thing twice, make it a workflow first.

### Working mode
Autonomous build–verify–iterate: design into the architecture → implement → write
adversarial tests that try to break it → run the test workflow → iterate to green → record
in PROGRESS.md. Verification is the approval; don't wait for sign-off on self-checkable steps.
