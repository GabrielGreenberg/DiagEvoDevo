# CLAUDE.md — Project Constitution

> This file is loaded at the start of every session. It is short on purpose.
> The substance lives in `CONCEPT.md` (the ideas) and `ARCHITECTURE.md` (the build).
> Current state always lives in `PROGRESS.md`. Resume/handoff protocol is `HANDOFF.md`.

## Mission

Build a **diagram evolver**: a system that starts from a random configuration of
line segments and, guided only by a principled scoring function, evolves it into a
faithful diagram of a small dataset. The long-run ambition is to *discover* diagram
kinds (bars, dot plots, radial plots, …) as optima of the score rather than to
hard-code any of them. The bar chart is the first target only because it is the
optimum we can check by hand.

The score is not a heuristic. It is a formal theory of when a figure is a good
**homomorphism** of its data. Treat the math as the product; the code serves it.

---

## The two cardinal principles

These are non-negotiable and are **reproduced verbatim in every handoff** so they
never erode across sessions.

### I. Depth over patches
Add every feature at the deepest correct layer. If a change belongs in the
architecture, change the architecture — refactor rather than bolt on. A surgical
patch that avoids a warranted structural change is a defect, not a shortcut. When
unsure whether a change is deep or shallow, choose deep, even at the cost of a larger
refactor now. We are building a conceptual instrument; its internal structure must
stay faithful to the ideas in `CONCEPT.md`, not accumulate convenience hacks.

### II. Workflows over re-derivation
Build and verify through repeatable workflows — scripts, task runners, test suites —
not by re-reading files or re-deriving results inside the context window. Prefer
invoking a workflow to reconstructing knowledge. This holds for **construction** as
much as for verification: if you find yourself doing the same multi-step thing twice,
make it a workflow first. This preserves context length and makes every result
reproducible on demand.

---

## How we work each session

1. **Read state, don't reconstruct it.** Start by reading `PROGRESS.md` and the most
   recent file in `handoffs/`. Do not re-derive prior decisions; they are recorded.
2. **Autonomous build–verify–iterate loop.** For each unit of work: design it into the
   architecture (Principle I) → implement → write *adversarial* tests that try to break
   it → run the test workflow → iterate until green → record in `PROGRESS.md`. Do not
   ask for approval between steps you can verify yourself; verification is the approval.
3. **Adversarial verification is mandatory.** A feature is not "done" because it runs.
   It is done when tests that actively attempt to falsify its invariants all pass.
   The invariant catalogue is in `ARCHITECTURE.md §Verification`.
4. **Keep the math modular.** Every scoring term, measurement, and weight is a
   registered, swappable unit with its parameters in `config`. The user will edit the
   math directly; never inline a constant that belongs in config.
5. **Log as you go.** Append decisions, done-items, and open questions to `PROGRESS.md`
   in the same session they occur. `PROGRESS.md` is the single source of truth for state.

## Handoff

When the user types **`hand off`**, follow `HANDOFF.md`: write a fresh, self-contained
resume prompt into `handoffs/`, re-embedding the two cardinal principles verbatim, and
stop. See `HANDOFF.md` for the exact contents required.
